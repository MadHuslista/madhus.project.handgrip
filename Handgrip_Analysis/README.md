# Handgrip Frequency Analysis Project

This project analyzes handgrip captures produced by the `LSL_Bridge` CSV logger and supports the staged workflow needed to:

1. characterize startup zero drift / warm-up bias,
2. characterize stationary noise,
3. characterize loaded drift / creep,
4. characterize real handgrip dynamics,
5. compare interference conditions, and
6. benchmark candidate digital filters before implementation.

The project is intentionally command-line driven and file-based so you can keep data capture independent from analysis.

---

## 1. Supported input format

The scripts expect CSV files generated by the current `LSL_Bridge` output format, with columns like:

- `host_unix_time_ns`
- `lsl_timestamp_s`
- `seq`
- `device_clock_us`
- `target_raw_count`
- `target_current_units`
- `target_filtered_units`
- `target_status`
- `raw_line`

Only `target_raw_count` is required. `target_filtered_units` is optional and used where available.

## 3. Setup

Create and activate a virtual environment, then install the package in editable mode.

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e .[dev]
```

All commands below assume you run them from the library root.

The project uses Hydra overrides, so arguments are passed as `name=value` (not `--name value`).

Input/output arguments are standard Hydra overrides in this project, so use `input=...`, `outdir=...`, `inputs=[...]`, `labels=[...]`, and related keys without a `+` prefix.

You can run each stage either with installed entry points (recommended) or via scripts:

- entry points: `ha-stage1`, `ha-stage2`, ..., `ha-run-all`
- scripts: `python scripts/stage1_startup_warmup.py`, etc.

## 4. Capture recommendations

The quality of the DSP conclusions depends on how captures are recorded. Keep **one CSV per condition / trial**.

### Recommended bridge-side settings during characterization

- Prefer **80 SPS** on the HX711 during characterization.
- Prefer logging the least processed signal available.
- Keep both `target_raw_count` and `target_filtered_units` in CSVs when possible.
- Do not change mechanics/cabling/mounting between trials unless that change is the test condition.
- Record condition metadata externally (ambient temperature, protocol, power source, etc.).

### File naming convention

Use descriptive names:

```text
YYYYMMDD_stage_condition_trial.csv
```

Examples:

```text
20260402_stage1_cold_start_trial01.csv
20260402_stage2_rest_after_warmup_trial01.csv
20260402_stage3_constant_load_20kg_trial01.csv
20260402_stage4_fast_max_grip_trial01.csv
20260402_stage5_usb_power_display_on_trial01.csv
```

### Capture manifest

Use a manifest CSV for batch execution with `ha-run-all`.

Required columns:

- `stage`
- `label`
- `path`

Optional columns:

- `channel`
- `notes`

## 5. What to capture for each stage

## Stage 1 - Startup warm-up / zero stabilization

Goal: quantify slow zero movement after power-on.

Capture protocol:
- Start from power-off / cold start.
- No hand contact, no load.
- Record continuously for **15 to 30 minutes**.
- Repeat at least 5 times if possible.

Run:

```bash
python scripts/stage1_startup_warmup.py analysis=stage1 \
  input=data/calibration_signals/20260402_stage1_cold_start_trial01.csv \
  outdir=data/analysis_results/stage_1 \
  analysis.channel=raw
```

What it produces:
- warm-up plots,
- rolling mean / std / slope,
- suggested ready time,
- summary JSON.

## Stage 2 - Static rest noise

Goal: characterize stationary noise after warm-up.

Capture protocol:
- Wait until thermally stable.
- No load, no hand contact.
- Record **10 to 20 minutes**.

Run:

```bash
python scripts/stage2_static_noise.py analysis=stage2 \
  input=data/calibration_signals/20260402_stage2_rest_after_warmup_trial01.csv \
  outdir=data/analysis_results/stage_2
```

What it produces:
- sampling stats,
- histogram,
- PSD,
- Allan deviation,
- dominant spectral peaks,
- raw vs filtered comparison when both channels exist.

## Stage 3 - Loaded drift / creep

Goal: separate no-load behavior from load-induced drift.

Capture protocol:
- After warm-up, apply a stable known load.
- Hold for **10 to 20 minutes**.
- Optionally include pre-load and post-unload windows in the same file.

Run:

```bash
python scripts/stage3_loaded_drift.py analysis=stage3 \
  input=data/calibration_signals/20260402_stage3_constant_load_4500g_trial01.csv \
  outdir=data/analysis_results/stage_3 \
  analysis.channel=raw
```

What it produces:
- drift slope,
- detrended view,
- return-to-zero estimate when pre/post windows are available,
- summary JSON.

## Stage 4 - Real handgrip dynamics

Goal: preserve true handgrip force-time structure while quantifying noise.

Recommended trial types:
- `fast_max`: squeeze as fast/hard as possible, hold 1-2 s, release.
- `ramp_hold`: ramp over ~1-2 s, hold 3-5 s, release.
- `sustained_hold`: fast squeeze, sustain 5-10 s, release.

Capture protocol:
- Record one file per trial.
- Include a few seconds of quiet baseline before grip.
- Repeat each trial type several times.

Run:

```bash
python scripts/stage4_grip_dynamics.py analysis=stage4 \
  'inputs=[data/calibration_signals/20260402_stage4_fast_max_trial01.csv,data/calibration_signals/20260402_stage4_ramp_hold_trial01.csv,data/calibration_signals/20260402_stage4_sustained_hold_trial01.csv]' \
  outdir=data/analysis_results/stage_4 \
  analysis.channel=raw
```

What it produces:
- detected grip-event windows,
- per-event metrics,
- overlays,
- hold-segment PSD,
- force derivative metrics.

## Stage 5 - Interference comparison

Goal: isolate deterministic contamination such as power-source coupling, display noise, or cable movement.

Capture protocol:
- Record one rest capture per condition.
- Change only one condition at a time.

Suggested conditions:
- battery vs USB power,
- display on vs off,
- cable fixed vs intentionally disturbed,
- BLE/radio on vs off,
- enclosure open vs closed.

Run:

```bash
python scripts/stage5_interference_compare.py analysis=stage5 \
  'inputs=[data/calibration_signals/20260402_stage4_fast_max_trial01.csv,data/calibration_signals/20260402_stage4_ramp_hold_trial01.csv,data/calibration_signals/20260402_stage4_sustained_hold_trial01.csv]' \
  'labels=[fast_max,ramp_hold,sustained_hold]' \
  outdir=data/analysis_results/stage_5 \
  analysis.channel=raw
```

What it produces:
- PSD overlays,
- dominant-peak comparison table,
- band-power comparison.

## Stage 6a - Filter design / candidate benchmark

Goal: compare candidate filters against noise reduction and waveform distortion.

Run:

```bash
python scripts/stage6_filter_design.py analysis=stage6 \
  input=data/calibration_signals/20260402_stage4_fast_max_trial01.csv \
  rest_input=data/calibration_signals/20260402_stage2_rest_after_warmup_trial01.csv \
  outdir=data/analysis_results/stage_6/design \
  analysis.filter_config=conf/filters/candidates.yaml
```

What it produces:
- candidate-filter comparison table,
- raw vs filtered overlays,
- PSD overlays,
- latency / peak-shift / rise-time change estimates when grip events exist.

## Stage 6b - Filter family review / ranking

Goal: rank candidate filters across one rest capture and multiple dynamic captures.

Run:

```bash
python scripts/stage6_filter_family_review.py analysis=stage6 \
  rest_input=data/calibration_signals/20260402_stage2_rest_after_warmup_trial01.csv \
  'dynamic_inputs=[data/calibration_signals/20260402_stage4_fast_max_trial01.csv,data/calibration_signals/20260402_stage4_ramp_hold_trial01.csv,data/calibration_signals/20260402_stage4_sustained_hold_trial01.csv]' \
  outdir=data/analysis_results/stage_6/review \
  analysis.filter_config=conf/filters/candidates.yaml
```

What it produces:
- `filter_family_assessment.csv` with per-filter metrics and composite score,
- `rest_psd_peaks.csv`,
- ranking plots (`composite_score.png`, `rest_psd_top_candidates.png`),
- summary JSON.

## 7. Important DSP interpretation notes

- **Startup drift is not random noise.** Treat it as baseline evolution.
- **Physiological variability is not always noise.** In handgrip tasks, low-frequency and tremor-band content can be true signal.
- Use the raw channel for system characterization whenever possible.
- Use filtered channels mainly for comparison.
- Avoid approving a filter only because the trace "looks smoother". Use stage 6 metrics.

## 8. Batch execution from a manifest

If you prepare a manifest CSV, you can run all stages that match the manifest rows:

```bash
ha-run-all \
  manifest=path/to/capture_manifest.csv \
  base_outdir=outputs/batch_run
```

Supported `stage` values in the manifest:

- `stage1`
- `stage2`
- `stage3`
- `stage4`
- `stage5`
- `stage6_design`
- `stage6_review`

The manifest runner dispatches rows by `stage` and forwards paths/channel settings to each stage script.
