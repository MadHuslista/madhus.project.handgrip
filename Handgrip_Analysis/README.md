# Handgrip Frequency Analysis Project

This project analyzes handgrip captures produced by the `LSL_Bridge` CSV logger and supports the staged workflow needed to:

1. characterize startup zero drift / warm-up bias,
2. characterize stationary noise,
3. characterize loaded drift / creep,
4. characterize real handgrip dynamics,
5. compare interference conditions, and
6. benchmark candidate digital filters before implementation.

The project is intentionally command-line driven and file-based so you can keep the data capture step independent from the analysis step.

---

## 1. Supported input format

The scripts expect CSV files generated by the current `LSL_Bridge` output format, with columns like:

- `host_unix_time_ns`
- `lsl_timestamp_s`
- `device_clock_us`
- `value_raw`
- `value_filtered`
- `sequence`
- `parser_mode`
- `raw_line`

The analysis scripts treat `value_raw` as the main characterization channel by default. `value_filtered` can also be analyzed for comparison against the placeholder bridge-side filter chain.

## 4. Capture recommendations

The quality of the DSP conclusions depends on how the captures are recorded. Use the bridge to generate CSVs and keep **one CSV per condition / trial**.

### Recommended bridge-side settings during characterization

- Prefer **80 SPS** on the HX711 during characterization.
- Prefer logging the **least processed signal available**.
- Keep both `value_raw` and `value_filtered` in the CSV so you can compare the current placeholder filter against the raw channel.
- Do not change sensor mechanics, cable routing, or mounting between trials unless that change is the condition under test.
- Write down condition metadata (ambient temperature, grip protocol, power source, etc.).

### File naming convention

Use a descriptive naming convention like:

```text
YYYYMMDD_stage_condition_trial.csv
```

Examples:

```text
20260402_stage1_cold_start_trial01.csv
20260402_stage2_rest_after_warmup_trial01.csv
20260402_stage3_constant_load_20kg_trial01.csv
20260402_stage4_fast_max_grip_trial01.csv
20260402_stage5_usb_power_display_on_trial01.csv
```

### Capture manifest

Use `data/capture_manifest_template.csv` as the starting point for a manifest. You can later feed that manifest to `scripts/run_all.py`.

Columns:

- `stage`
- `label`
- `path`
- `channel`
- `notes`

## 5. What to capture for each stage

## Stage 1 — Startup warm-up / zero stabilization

Goal: quantify the slow zero movement after power-on.

Capture protocol:
- Start from power-off / cold start.
- No hand contact, no load.
- Record continuously for **15 to 30 minutes**.
- Repeat at least 5 times if possible.
- Record ambient temperature if available in external notes.

Run:

```bash
python scripts/stage1_startup_warmup.py   --input path/to/cold_start.csv   --channel raw   --outdir outputs/stage1_cold_start_trial01
```

What it produces:
- warm-up plots,
- rolling mean / std / slope,
- suggested ready time,
- summary JSON.

## Stage 2 — Static rest noise

Goal: characterize stationary noise after warm-up.

Capture protocol:
- Wait until the system is thermally stable.
- No load, no hand contact.
- Record **10 to 20 minutes**.
- Prefer a mechanically quiet environment.

Run:

```bash
python scripts/stage2_static_noise.py   --input path/to/rest_after_warmup.csv   --outdir outputs/stage2_rest_trial01
```

What it produces:
- sampling stats,
- histogram,
- PSD,
- Allan deviation,
- dominant spectral peaks,
- raw vs filtered comparison when both channels exist.

## Stage 3 — Loaded drift / creep

Goal: separate no-load behavior from load-induced drift.

Capture protocol:
- After warm-up, apply a stable known load.
- Hold for **10 to 20 minutes**.
- Optionally include pre-load and post-unload windows in the same file.
- Repeat for several load levels.

Run:

```bash
python scripts/stage3_loaded_drift.py   --input path/to/constant_load.csv   --channel raw   --outdir outputs/stage3_load20kg_trial01
```

What it produces:
- drift slope,
- detrended view,
- return-to-zero estimate when pre/post windows are available,
- summary JSON.

## Stage 4 — Real handgrip dynamics

Goal: preserve the true handgrip force-time structure while quantifying noise.

Recommended trial types:
- `fast_max`: squeeze as fast and hard as possible, hold 1–2 s, release.
- `ramp_hold`: ramp over ~1–2 s, hold 3–5 s, release.
- `sustained_hold`: fast squeeze, sustain 5–10 s, release.

Capture protocol:
- Record one file per trial.
- Include at least a few seconds of quiet baseline before the grip.
- Repeat each trial type several times.

Run:

```bash
python scripts/stage4_grip_dynamics.py   --input path/to/fast_max_trial01.csv path/to/fast_max_trial02.csv   --channel raw   --outdir outputs/stage4_fast_max
```

What it produces:
- detected grip-event windows,
- per-event metrics,
- overlays,
- hold-segment PSD,
- force derivative metrics.

## Stage 5 — Interference comparison

Goal: isolate deterministic contamination such as power-source coupling, display noise, or cable movement.

Capture protocol:
- Record one rest capture per condition.
- Change only one condition at a time.

Suggested conditions:
- battery vs USB power,
- display on vs off,
- cable fixed vs intentionally disturbed,
- BLE/radio on vs off,
- enclosure open vs closed.

Run:

```bash
python scripts/stage5_interference_compare.py   --inputs cond_a.csv cond_b.csv cond_c.csv   --labels battery usb display_on   --channel raw   --outdir outputs/stage5_compare
```

What it produces:
- PSD overlays,
- dominant-peak comparison table,
- band-power comparison.

## Stage 6 — Filter design / candidate benchmark

Goal: compare candidate filters against noise reduction and waveform distortion.

Run:

```bash
python scripts/stage6_filter_design.py   --input path/to/trial.csv   --channel raw   --config configs/filter_candidates.yaml   --outdir outputs/stage6_filter_design
```

What it produces:
- candidate-filter comparison table,
- raw vs filtered overlays,
- PSD overlays,
- latency / peak-shift / rise-time change estimates when grip events exist.

## 7. Important DSP interpretation notes

- **Startup drift is not the same as random noise.** Treat it as baseline evolution.
- **Physiological variability is not always noise.** In handgrip tasks, some low-frequency and tremor-band content is part of the real output.
- Use the raw channel for system characterization whenever possible.
- Use the filtered channel mainly as a comparison baseline against the current bridge-side placeholder filter.
- Avoid approving a filter only because the trace “looks smoother.” Use the stage 6 metrics instead.

---

## 8. Batch execution from a manifest

If you prepare a manifest CSV, you can run all stages that match the manifest rows:

```bash
python scripts/run_all.py   --manifest data/capture_manifest_template.csv   --base-outdir outputs/batch_run
```

The manifest runner is intentionally simple. It dispatches rows by the `stage` column and forwards the file paths to the stage scripts.
